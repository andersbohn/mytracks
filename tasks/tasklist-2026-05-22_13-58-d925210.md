## Updating the VPS

### First-time setup

Before the first deploy, run the steps in `yaml/InstallOnVps.md` on the VPS. This creates the `mytracks` database and user inside the existing picklist postgres pod, then creates the `mytracks-secret` Kubernetes secret with your DB credentials, Google OAuth2 client ID/secret, and allowed-email list.

Once the secret exists, apply the ingress manually (the CI only redeploys the app manifest, not the ingress):

```shell
kubectl apply -f yaml/traefik-ingress.yaml
```

This only needs to be done once. Subsequent deploys via CI will not touch the ingress.

### Routine deploys (CI-driven)

Pushing a version tag triggers the `deploy` job automatically:

```shell
git tag 1.0.0
git push origin 1.0.0
```

CI builds a Docker image, pushes it to `ghcr.io/<owner>/mytracks:1.0.0`, and applies the updated `yaml/app-deployment.yaml` to the cluster via SSH. The deployment rolls out with the new image; k3s waits for the readiness probe before cutting over traffic.

### Manual re-deploy (no code change)

To restart the running pod without a new tag — e.g. after rotating a secret — run on the VPS:

```shell
kubectl rollout restart deployment/mytracks
kubectl rollout status deployment/mytracks
```

### Updating the ingress

If ingress rules change (e.g. adding a new path or middleware), apply the updated file directly on the VPS:

```shell
kubectl apply -f yaml/traefik-ingress.yaml
```

The CI does not manage the ingress, so this is always a manual step.

### Rotating secrets

Update the k8s secret in place, then restart the deployment to pick up the new values:

```shell
kubectl create secret generic mytracks-secret \
  --from-literal=db-username=mytracks \
  --from-literal=db-password=NEW_PASSWORD \
  --from-literal=oauth-client-id=CLIENT_ID \
  --from-literal=oauth-client-secret=CLIENT_SECRET \
  --from-literal=allowed-emails=you@example.com \
  --save-config --dry-run=client -o yaml | kubectl apply -f -
kubectl rollout restart deployment/mytracks
```
